## 1. 수집하는 개인정보

본 서비스는 임직원의 업무 수행을 지원하기 위해 최소한의 개인정보만을 수집합니다.

### 필수 수집 정보

```
필수 수집 정보
├── 계정 정보
│   ├── 이메일 주소 (식별자, SSO 연동 기준)
│   └── 이름 (표시 이름)
│
└── 서비스 이용 정보
    ├── 할일(Todo) 내용, 생성일, 마감일, 우선순위
    ├── 카테고리(Category) 정보
    ├── 태그(Tag) 정보
    └── 기술 정보
        ├── Access Token (JWT, 인증 목적)
        └── Refresh Token (JWT, 세션 갱신 목적)
```

### 자동 수집 정보

```
자동 수집 정보
├── 서버 로그
│   ├── API 요청 시각 및 엔드포인트
│   ├── 클라이언트 IP 주소
│   └── 오류 발생 정보 (에러 코드, 스택 트레이스 제외)
│
└── 리마인더 발송 기록
    ├── 발송 대상 사용자 식별자
    ├── 발송 시각
    └── 발송 성공/실패 여부
```

### 수집하지 않는 정보

```
수집하지 않는 정보 (명시적 비수집 항목)
├── 비밀번호 (SSO 방식으로 인증하며 비밀번호를 직접 저장하지 않음)
├── 결제 정보 (사내 도구로 결제 기능 없음)
├── 위치 정보 (GPS 또는 IP 기반 정밀 위치 추적 없음)
└── 생체 정보 (지문, 얼굴 인식 등 일체 수집 없음)
```

## 2. 개인정보 수집 목적

본 서비스는 다음의 목적을 위해서만 개인정보를 처리하며, 목적 외 용도로 사용하지 않습니다.

| 처리 목적 | 세부 내용 | 처리 근거 |
|---|---|---|
| **서비스 제공 — 인증** | JWT 토큰 발급 및 검증을 통해 임직원 본인 확인 및 세션 관리 | 계약(근로 관계) 이행 |
| **서비스 제공 — 할일 관리** | 할일 생성·조회·수정·삭제, 카테고리·태그 관리, 우선순위 설정 | 계약(근로 관계) 이행 |
| **리마인더 발송** | 마감일 기반 리마인더 알림 발송 및 발송 이력 기록 | 계약(근로 관계) 이행 |
| **서비스 개선 및 오류 추적** | 서버 로그 분석을 통한 버그 수정, 성능 개선, 비정상 접근 탐지 | 정당한 이익 (사내 IT 시스템 안정 운영) |

> **주의:** 개인정보는 위 목적 이외에 마케팅, 프로파일링, 외부 제공 등의 용도로 사용되지 않습니다.

## 3. 개인정보 보관 기간

### 데이터 유형별 보관 기간

| 데이터 유형 | 보관 기간 | 만료 후 처리 |
|---|---|---|
| **사용자 계정** (User: id, email, name) | 재직 기간 전체 + 퇴직 후 30일 | 영구 삭제 (DB row 물리 삭제) |
| **정상 상태 할일** (Todo: 삭제되지 않은 항목) | 계정 활성 기간 무기한 보관 | 계정 삭제 시 함께 삭제 |
| **삭제된 할일** (소프트 딜리트 항목) | 소프트 딜리트 처리일로부터 30일 | 30일 경과 후 DB에서 물리 삭제 |
| **카테고리 / 태그** | 계정 활성 기간 무기한 | 계정 삭제 시 함께 삭제 |
| **서버 로그** | 생성일로부터 90일 | 90일 경과 후 자동 삭제 또는 파기 |
| **리마인더 발송 기록** | 발송일로부터 6개월 | 6개월 경과 후 자동 삭제 |
| **Access Token** | 발급 후 15분 (토큰 자체 만료) | 만료 후 자동 무효화 |
| **Refresh Token** | 발급 후 7일 (토큰 자체 만료) | 만료 후 자동 무효화 |

### 퇴직자 데이터 처리 절차

퇴직이 확정된 임직원의 데이터는 아래 절차에 따라 처리합니다.

1. **퇴직 확인:** HR 시스템 또는 담당 관리자로부터 퇴직 일자 통보 수령
2. **즉시 조치 (퇴직일 당일):** 계정 비활성화 처리 — 로그인 차단, 신규 토큰 발급 중단
3. **30일 유예 기간:** 업무 인수인계에 필요한 할일·카테고리·태그 데이터 조회 가능 상태 유지 (관리자 권한으로만 접근)
4. **퇴직 후 30일:** 사용자 계정 및 연관 데이터 전체 영구 삭제

## 4. 개인정보 접근 및 처리 주체

### 시스템 자동 처리

본 서비스의 백엔드(Node.js + Express + Prisma)는 사용자 요청에 따라 자동으로 개인정보를 처리합니다. 자동 처리 항목은 인증 토큰 검증, 할일 CRUD 연산, 리마인더 스케줄 실행, 서버 로그 기록입니다.

### 개발팀 접근 범위

- **DB 직접 접근 권한 보유자:** 프로덕션 DB(PostgreSQL)에 직접 접속할 수 있는 인원은 최소화하며, 반드시 사전 승인된 개발팀 담당자로 제한합니다.
- **접근 로그 기록:** DB 직접 접근 시 접속 일시, 접속자, 수행 쿼리 요약을 기록합니다.
- **목적 외 조회 금지:** 개발·운영·장애 대응 목적 이외에 개인 업무 내용을 열람하는 행위를 금지합니다.
- **최소 권한 원칙:** 개발자별로 업무에 필요한 최소한의 DB 권한만 부여합니다.

### 제3자 제공 금지 원칙

본 서비스에서 수집된 개인정보는 어떠한 경우에도 외부 제3자(협력사, 광고 플랫폼, 타 기관 등)에 제공하지 않습니다. 단, 법령에 따른 수사기관의 적법한 요청이 있는 경우는 예외로 하며, 이 경우 정보주체에게 사전 또는 사후 고지합니다.

### 외부 서비스 연동 시 고지 의무

향후 이메일 발송, 슬랙 알림, SSO 제공자 등 외부 서비스와의 연동이 추가될 경우, 해당 서비스의 명칭, 제공 데이터 범위, 처리 목적을 본 정책에 명시하고 임직원에게 사전 고지합니다.

> **현재 연동 현황 (v1):** 외부 서비스 연동 없음. 이메일 발송 기능은 현재 Mock 처리 중이며 실제 외부 전송 없음.

## 5. 기술적 보호 조치

### 현재 구현된 보호 조치

```
기술적 보호 조치
│
├── 전송 구간 보호
│   ├── HTTPS 필수 적용 (프로덕션 환경)
│   └── JWT 서명 검증 (모든 API 요청에서 토큰 위변조 검증)
│
├── 저장 데이터 보호
│   ├── 데이터 사용자 단위 격리 (userId 기반 조회 — 타인 데이터 접근 불가)
│   └── DB 접근 권한 최소화 (Prisma ORM 경유, 직접 접속 제한)
│
└── 인증 / 인가
    ├── Access Token 만료 시간: 발급 후 15분
    ├── Refresh Token 만료 시간: 발급 후 7일
    ├── 모든 보호 API에 requireAuth 미들웨어 적용
    └── 타인 리소스 접근 시도 시 HTTP 403 반환
```

### 현재 미비 사항 (v1 — 개선 필요)

아래 항목은 현재 버전(v1)에서 아직 구현되지 않았으며, 보안 강화를 위해 우선적으로 개선이 필요합니다.

| 미비 항목 | 현재 상태 | 위험 내용 | 권장 조치 |
|---|---|---|---|
| **JWT 토큰 저장 방식** | 브라우저 `localStorage`에 저장 | XSS 공격 시 토큰 탈취 가능 | `httpOnly` + `Secure` 쿠키 방식으로 전환 |
| **Refresh Token 강제 회수** | 서버 측 블랙리스트 미구현 | 탈취된 Refresh Token을 7일간 회수 불가 | 서버 측 토큰 블랙리스트 또는 DB 기반 토큰 관리 구현 |
| **전체 감사 로그(Audit Log)** | 미구현 | 데이터 접근·변경 이력 추적 불가 | 주요 데이터 변경(생성·수정·삭제) 이벤트에 대한 Audit Log 테이블 구현 |

> 위 미비 사항은 본 서비스 v2 로드맵에 포함되어 있으며, 사내 시스템인 점을 감안하더라도 조속한 개선이 권장됩니다.

## 7. 퇴직자 처리 절차

퇴직자의 개인정보는 업무 연속성 보장과 데이터 보호 원칙 사이의 균형을 고려하여 아래 절차에 따라 처리합니다.

### 처리 절차 상세

**1단계 — 퇴직 확인 시점**
- HR 부서로부터 퇴직 예정일 공식 통보를 수령하거나, 고용 계약 만료일을 기준으로 합니다.
- 비자발적 퇴직(해고, 계약 해지)의 경우 해당 일자를 퇴직 확인 시점으로 간주합니다.

**2단계 — 계정 비활성화 (퇴직일 당일)**
- 대상 계정에 대한 로그인 요청을 즉시 차단합니다.
- 유효한 Access Token 및 Refresh Token을 무효화합니다.
- 계정 상태를 `inactive`로 변경합니다 (DB 레코드는 유지).

**3단계 — 유예 기간 데이터 보존 (퇴직 후 최대 30일)**
- 업무 인수인계를 위해 해당 사용자의 할일·카테고리·태그 데이터를 관리자 권한으로 열람 가능한 상태로 유지합니다.
- 이 기간 중 해당 데이터는 퇴직자 본인은 접근할 수 없으며, 승인된 팀 관리자만 인수인계 목적으로 접근합니다.

**4단계 — 완전 삭제 (퇴직 후 30일 경과)**
- 사용자 계정 레코드(User 테이블) 물리 삭제
- 연관된 모든 할일(Todo), 카테고리(Category), 태그(Tag) 데이터 물리 삭제
- 리마인더 기록은 별도 보관 기간(6개월) 경과 후 삭제
- 서버 로그는 별도 보관 기간(90일) 경과 후 삭제
- 삭제 수행 내역을 처리 대장에 기록 (처리자, 처리 일시, 삭제 데이터 범위)

## 8. 개인정보 침해 대응

### 침해 인지 시 내부 보고 경로

개인정보 침해가 의심되거나 확인된 경우, 인지한 구성원은 즉시 아래 경로로 보고합니다.

```
침해 인지 구성원
    └── 즉시 보고 (1시간 이내)
         └── 개발팀 보안 담당자 / 팀장
              └── 경영진 / 정보보호 책임자 보고 (4시간 이내)
                   └── 침해 범위 및 영향 조사 착수
```

### 사용자 통보 기준

- **통보 대상:** 개인정보가 유출·훼손·변조되었거나 그럴 가능성이 있는 경우
- **통보 시한:** 침해 사실 인지 후 **72시간 이내** 영향을 받은 임직원에게 고지
- **통보 방법:** 사내 이메일, 서비스 내 공지, 또는 직접 연락
- **통보 내용:** 침해 발생 일시, 침해된 정보의 항목, 침해 경위, 피해 최소화를 위해 취할 수 있는 조치, 담당자 연락처

### 사후 조치

1. **긴급 조치:** 침해 경로 차단, 영향받은 계정 즉시 접근 제한
2. **원인 분석:** 침해 경위 및 취약점 조사, 로그 분석
3. **재발 방지:** 취약점 패치, 보안 정책 업데이트, 필요 시 외부 보안 전문가 자문
4. **결과 보고:** 경영진 보고 및 재발 방지 대책 수립·이행

## 9. 정책 변경 및 고지

### 정책 변경 시 사전 고지

본 개인정보 보호 정책은 서비스 변경, 법령 개정, 또는 내부 정책 변화에 따라 업데이트될 수 있습니다.

- **경미한 변경** (오탈자 수정, 연락처 변경 등): 정책 문서 업데이트 후 변경 일자 명시
- **중요한 변경** (수집 항목 추가, 보관 기간 변경, 처리 목적 변경 등): 변경 시행일로부터 **최소 7일 전** 사내 공지 채널(이메일, 인트라넷 등)을 통해 사전 고지

### 중요 변경 시 재동의 절차

수집 항목의 신규 추가, 처리 목적 변경, 제3자 제공 추가 등 정보주체의 권리에 중대한 영향을 미치는 변경이 발생하는 경우, 서비스 내 동의 절차 또는 서면을 통해 임직원의 재동의를 받습니다.

재동의를 거부하는 경우, 해당 항목의 처리를 수행하지 않으며 이로 인해 서비스 이용이 제한될 수 있습니다.

## 10. GDPR / 개인정보보호법 준수 체크리스트

본 체크리스트는 국내 「개인정보 보호법」 및 GDPR(General Data Protection Regulation)의 주요 요건에 대한 현재 이행 상태를 나타냅니다.

| 항목 | 요건 내용 | 현재 상태 | 조치 필요 여부 |
|---|---|---|---|
| **처리 근거 명시** | 개인정보 처리 시 적법한 근거(동의, 계약 이행, 정당한 이익 등) 확보 및 명시 | 부분 이행 — 본 정책에 명시하였으나 서비스 내 동의 UI 없음 | 필요 — 서비스 최초 로그인 시 정책 확인 및 동의 UI 추가 권장 |
| **최소 수집 원칙** | 처리 목적에 필요한 최소한의 정보만 수집 | 이행 — 이메일·이름·업무 데이터만 수집, 불필요 항목 수집 없음 | 불필요 |
| **보관 기간 제한** | 필요 기간을 초과하여 개인정보를 보관하지 않음 | 부분 이행 — 정책에 기간 명시 완료, 자동 삭제 로직 구현 필요 | 필요 — 만료 데이터 자동 삭제 배치 작업 구현 |
| **정보주체 권리 보장** | 열람·정정·삭제·이동·처리 제한 권리 행사 절차 마련 | 부분 이행 — 열람·정정·삭제 일부 지원, 이동권·계정 삭제 미지원 | 필요 — v2에서 계정 삭제 및 데이터 Export 기능 구현 |
| **위반 신고 절차** | 침해 발생 시 72시간 내 통보 절차 마련 | 이행 — 본 정책에 절차 명시 완료 | 절차 문서화 완료, 실제 훈련 및 담당자 지정 필요 |
| **기술적 보호 조치** | 암호화, 접근 통제 등 기술적 안전 조치 적용 | 부분 이행 — HTTPS, JWT, 사용자 격리 적용 / localStorage 저장, Audit Log 미비 | 필요 — httpOnly 쿠키 전환, Audit Log, 토큰 블랙리스트 구현 |
| **개인정보 처리 방침 공개** | 처리 방침을 정보주체가 쉽게 확인할 수 있도록 공개 | 부분 이행 — 정책 문서 작성 완료, 서비스 내 접근 경로 없음 | 필요 — 앱 내 개인정보 처리방침 링크 추가 |
| **수탁자 관리** | 개인정보 처리를 외부에 위탁하는 경우 수탁자 관리 의무 이행 | 해당 없음 (현재 외부 위탁 없음) | 외부 서비스 연동 추가 시 수탁자 계약 및 고지 의무 발생 |
| **국외 이전 제한** | 개인정보 국외 이전 시 요건 충족 필요 | 해당 없음 (현재 국내 서버 운영 가정) | 해외 클라우드 서비스 도입 시 별도 검토 필요 |
| **책임자 지정** | 개인정보 보호 책임자(CPO) 또는 담당자 지정 | 미이행 — 공식 담당자 미지정 | 필요 — 정보보호 담당자 지정 및 연락처 공개 |
