## 3. 상태 머신 (Status State Machine)

├── 3.1 상태 정의
│   ├── 3.1.1 TODO — 기본 대기 상태 [구현됨]
│   ├── 3.1.2 IN_PROGRESS — 진행 중 [구현됨]
│   ├── 3.1.3 DONE — 완료 [구현됨]
│   └── 3.1.4 DELETED — 소프트 딜리트 [구현됨]
│
├── 3.2 허용 전이
│   ├── 3.2.1 TODO → IN_PROGRESS (작업시작) [구현됨]
│   ├── 3.2.2 TODO → DELETED (삭제) [구현됨]
│   ├── 3.2.3 IN_PROGRESS → DONE (체크박스) [구현됨]
│   ├── 3.2.4 IN_PROGRESS → TODO (보류) [구현됨]
│   ├── 3.2.5 IN_PROGRESS → DELETED (삭제) [구현됨]
│   ├── 3.2.6 DONE → TODO (완료 취소) [구현됨]
│   └── 3.2.7 DELETED → TODO (복원) [구현됨]
│
├── 3.3 금지 전이 (422 반환)
│   ├── 3.3.1 TODO → DONE 직접 전이 금지 [구현됨]
│   ├── 3.3.2 DONE → DELETED 직접 전이 금지 [구현됨]
│   └── 3.3.3 DONE → IN_PROGRESS 직접 전이 금지 [구현됨]
│
└── 3.4 사이드 이펙트
    ├── 3.4.1 → DONE 전환 시
    │   └── 3.4.1.1 미발송 리마인더 isSent = true 일괄 처리 [구현됨]
    │
    ├── 3.4.2 DONE → TODO 전환 시
    │   └── 3.4.2.1 마감일 미경과 시 리마인더 재생성 [구현됨]
    │
    ├── 3.4.3 → DELETED 전환 시
    │   ├── 3.4.3.1 미발송 리마인더 isSent = true 일괄 처리 [구현됨]
    │   └── 3.4.3.2 deletedAt 타임스탬프 기록 [구현됨]
    │
    └── 3.4.4 DELETED → TODO 전환 시 (복원)
        ├── 3.4.4.1 isDeleted = false, deletedAt = null 초기화 [구현됨]
        └── 3.4.4.2 마감일 미경과 시 리마인더 재생성 [구현됨]

## v2 우선순위 목록 (ranked)

1. **Google Workspace SSO 연동** — 내부 도구 보안 및 온보딩의 핵심. 도메인 제한 (@saladlab.com) 포함. 데모 로그인 교체.
2. **httpOnly Cookie 세션 전환** — XSS 공격으로부터 토큰을 보호. SSO 전환과 병행 작업 가능.
3. **Refresh Token Rotation** — Access Token 자동 갱신은 구현됐으나 Refresh Token 탈취 시 무력화 방어가 없음.
4. **서버 사이드 토큰 블랙리스트** — 로그아웃 즉시 무효화가 불가능한 Stateless 구조 개선.
5. **실제 이메일 발송 연동 (SMTP / SES)** — 리마인더 기능의 실사용 가치를 완성. console.log mock을 실 발송으로 교체.
6. **태그 단독 필터링** — 태그 입력 경로는 완성됐으나 필터 활용이 막혀 UX 불완전.
7. **발송 실패 재시도 로직 (Cron)** — 리마인더 신뢰성 향상. 이메일 연동 이후 자연스럽게 추가 가능.
8. **30일 자동 영구 삭제 배치 (purgeWorker)** — 정책 설계 완료. 실행 로직만 추가하면 완성 가능.

## No-Gos

아래 항목은 현재 로드맵에서 명시적으로 제외된다. 재논의가 필요하다면 별도 ADR(Architecture Decision Record)을 작성한다.

| 항목 | 이유 |
|---|---|
| 실시간 협업 (멀티 유저 동시 편집) | WebSocket 도입 및 충돌 해소 로직 필요. 아키텍처 변경 범위 초과. |
| 외부 캘린더 동기화 (Google Calendar, Outlook) | OAuth 스코프 확장 및 양방향 동기화 복잡도 높음. |
| AI 자동 우선순위 제안 | 데이터 충분성 및 개인정보 처리 정책 미비. v2 이후 별도 검토. |
| 팀 / 조직 단위 공유 할일 | RBAC 및 멀티 테넌시 설계 없이 추가 불가. 별도 BRD 필요. |
| 모바일 네이티브 앱 (iOS / Android) | 웹 반응형으로 커버. 네이티브 전환은 별도 프로젝트. |
| 오프라인 모드 / PWA Service Worker | 로컬 캐시 전략과 충돌 해소 로직 미설계. ROI 낮음. |
| 타임 트래킹 | 핵심 가치(할일 완료)와 무관. 별도 도구 활용. |
| 캘린더 뷰 | 범위 초과. 기존 캘린더 도구 연동으로 대체. |
| 위키 / 문서 관리 | 기존 Notion 활용. 중복 도구 지양. |
| 다국어 (i18n) | 사내 한국어 단일 환경. 필요 시 별도 검토. |

## 기능 추가 판단 기준

새 기능 요청이 들어왔을 때 아래 기준으로 판단한다. 모든 항목이 YES여야 v1 패치 또는 v2 후보로 등록하고, 하나라도 NO이면 No-Gos 목록 검토 또는 다음 배치로 분리한다.

1. **사용자 가치** — 현재 내부 사용자 중 5인 이상이 직접 요청했거나, 기존 핵심 기능의 완성도를 직접 높이는가?
2. **범위 적합성** — 할일 / 작업 관리 핵심 루프 (생성 → 진행 → 완료) 안에 속하는가?
3. **구현 비용** — 기존 아키텍처 변경 없이 1스프린트(2주) 이내 완료 가능한가?
4. **데이터 정합성** — 상태 머신 전이 규칙, 소프트 딜리트 정책, 리마인더 사이드 이펙트와 충돌하지 않는가?
5. **테스트 가능성** — 유닛 테스트 또는 E2E 시나리오로 검증 가능한가?
6. **No-Gos 비해당** — 위 No-Gos 목록에 해당하지 않는가?

> 판단 결과가 v2 이상이라면 이 문서의 해당 도메인 트리에 `[v2]` 또는 `[미구현]` 노드로 먼저 등록한 후 개발을 진행한다.
